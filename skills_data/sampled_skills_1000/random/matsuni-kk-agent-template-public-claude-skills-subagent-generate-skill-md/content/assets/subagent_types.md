# サブエージェントタイプ一覧

## 1. research-agent（調査系）

### 用途
- 最新ドキュメント・ベストプラクティスの調査
- 技術動向のリサーチ
- 公式仕様の確認

### 典型的なタスク
- 「Slack SDK の最新APIを調査」
- 「Block Kit のベストプラクティスを収集」
- 「競合ライブラリの比較調査」

### 必須ツール
- WebSearch: Web検索
- WebFetch: ページ内容取得
- Read: ローカルファイル参照

### 携帯Skills例
```yaml
携帯Skills:
  - external-api: API仕様の理解
  - manifest: 設定項目の理解
```

### 出力フォーマット
```markdown
## 調査結果

### 概要
{1-2文の要約}

### 発見事項
1. {発見1}
   - ソース: {URL}
   - 取得日: {日付}

### 推奨事項
- {推奨1}
- {推奨2}
```

## 2. feedback-agent（フィードバック系）

### 用途
- コード・成果物の客観的レビュー
- 改善提案の生成
- ベストプラクティスとの比較

### 典型的なタスク
- 「Function実装をレビュー」
- 「manifest.tsの設定を確認」
- 「ワークフロー設計をレビュー」

### 必須ツール
- Read: 対象ファイル読み込み
- Grep: パターン検索
- Glob: ファイル探索

### 携帯Skills例
```yaml
携帯Skills:
  - custom-function: Function実装基準
  - error-handling: エラーハンドリング基準
  - testing: テストカバレッジ基準
```

### 出力フォーマット
```markdown
## レビュー結果

### 総合評価: {A/B/C/D}

### Good Points
- {良い点1}
- {良い点2}

### Improvements
| 重要度 | 箇所 | 指摘 | 改善案 |
|--------|------|------|--------|
| 高 | {location} | {issue} | {suggestion} |

### Next Actions
1. {アクション1}
2. {アクション2}
```

## 3. ideation-agent（アイデア出し系）

### 用途
- 複数の設計アプローチ提案
- ブレインストーミング
- 代替案の生成

### 典型的なタスク
- 「データ構造の設計案を3つ提案」
- 「ワークフローのアプローチを比較」
- 「UI設計のバリエーションを生成」

### 必須ツール
- Read: 既存コード参照
- WebSearch: 事例調査

### 携帯Skills例
```yaml
携帯Skills:
  - workflow: ワークフロー設計知識
  - datastore: データ設計知識
  - block-kit: UI設計知識
```

### 出力フォーマット
```markdown
## アイデア一覧

### 案1: {名前}
- **概要**: {説明}
- **Pros**: {利点}
- **Cons**: {欠点}
- **適用場面**: {いつ使うか}

### 案2: {名前}
...

### 推奨案
**{案X}** を推奨。理由: {理由}
```

## 4. validation-agent（検証系）

### 用途
- 設計・仮説の妥当性検証
- 技術的制約のチェック
- 実現可能性の評価

### 典型的なタスク
- 「このAPI設計は実現可能か検証」
- 「スコープ設定の妥当性を確認」
- 「外部API制約をチェック」

### 必須ツール
- Read: 仕様確認
- WebSearch: 制約調査
- Bash: 動作確認（必要時）

### 携帯Skills例
```yaml
携帯Skills:
  - manifest: 権限・制約の理解
  - external-api: API制約の理解
  - deploy: デプロイ制約の理解
```

### 出力フォーマット
```markdown
## 検証結果

### 判定: {可能/条件付き可能/不可能}

### 検証項目
| 項目 | 結果 | 備考 |
|------|------|------|
| {項目1} | ✅/⚠️/❌ | {備考} |

### リスク
- {リスク1}: {影響度}

### 前提条件
- {前提1}
- {前提2}
```

---

## 5. parallel-check-agent（並列チェック系）

### 用途
- 複数ファイルの同時チェック
- プロジェクト全体の一括検証
- 並列処理による高速化

### 典型的なタスク
- 「全Functionsのエラーハンドリングをチェック」
- 「全Workflowsのステップ構成を検証」
- 「全Triggersの設定を確認」

### 必須ツール
- Glob: ファイル一覧取得
- Read: 各ファイル読み込み
- Grep: パターンマッチ

### 携帯Skills例
```yaml
携帯Skills:
  - 対象ドメインのSkill全て
```

### 出力フォーマット
```markdown
## 並列チェック結果

### サマリー
- 対象ファイル数: {N}
- OK: {n}
- Warning: {n}
- Error: {n}

### 詳細
| ファイル | 結果 | 指摘 |
|----------|------|------|
| {file1} | ✅ | - |
| {file2} | ⚠️ | {issue} |

### 共通の問題
- {複数ファイルで見られる問題}
```
