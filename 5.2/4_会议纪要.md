# 会议纪要

## 一、会议主题

围绕 skill 场景下的恶意能力检测绕过问题，讨论现有 jailbreak 方法是否适用，并重新明确后续研究方向、攻击建模和实验路径。

## 二、核心结论

1. 当前问题更接近对 LLM-based 检测器或 judge 的规避攻击，传统 jailbreak 只能作为对比背景。
2. 传统 jailbreak 主要关注让模型突破自身安全约束并输出有害内容；本项目关注让检测工具误判恶意 skill、恶意 instruction 或恶意代码为 benign。
3. 后续应优先调研和复用 adversarial attack、LLM judge 欺骗、代码漏洞检测器规避等方向的方法。
4. 研究目标应重新表述为：评估现有已知规避方法能否绕过基于大模型的 skill 恶意检测器。
5. skill 架构总结等部分相对现成，关键突破点在攻击方法与实验验证。

## 三、讨论要点

### 1. 不建议直接用大模型生成攻击实现

与会者认为，直接让大模型根据 instruction 生成攻击样例会受到模型内置安全机制影响，不同模型的拒答策略也会造成明显偏差。例如部分模型会拒绝生成恶意代码，部分模型可能给出可执行攻击样例。因此，不宜把大模型作为该部分攻击实现的主要来源。

建议改为参考已有开源方法和论文，将其攻击形式改造到 skill 场景中。

### 2. jailbreak 与当前任务的差异

传统 jailbreak 的典型目标是设计 prompt，使被测模型突破系统提示词或安全对齐限制，输出恶意代码、色情暴力内容等有害结果。成功标准通常是模型输出了被禁止的内容，失败标准是模型拒绝回答。

本项目的目标是让检测阶段无法识别恶意 skill 或恶意 instruction。检测工具通常会调用 LLM 判断内容是否 malicious，项目关注的是如何使该判断出错。

因此，当前任务可建模为恶意内容识别或代码恶意性识别中的规避问题。

### 3. 更匹配的研究方向

会议认为，更相关的方向包括：

1. adversarial attack on LLM judge；
2. black-box adversarial attack；
3. LLM-based vulnerability detector 的规避攻击；
4. 针对代码检测器的变量名混淆、结构混淆、dead code 插入、编码与解码伪装等方法；
5. 将恶意功能拆分到多个 skill 中，通过连续执行形成恶意行为。

其中，欺骗 LLM judge 使其将有害内容判断为无害，与当前任务高度接近。

### 4. 可尝试的攻击形式

会议中提到若干可能的攻击路径：

1. 将原本写在 markdown instruction 中的恶意内容转移到代码中；
2. 对代码进行变量名混淆、结构混淆、编码伪装等处理；
3. 插入无关代码或干扰性特征，使 LLM-based 检测器误判；
4. 将单一恶意功能拆分为多个单独看似正常的 skill，通过组合执行形成恶意行为；
5. 利用邮件等外部输入作为间接指令来源，测试 agent 是否会执行伪装指令；
6. 参考漏洞检测器规避论文，将“有漏洞代码检测”类方法迁移到“有攻击意图代码检测”场景。

### 5. 实验目标

后续实验应围绕 evaluation 展开，即系统测试现有已知攻击或伪装方法能否绕过当前 LLM-based 检测器。

重点不在于提出全新攻击算法，而在于：

1. 找到与当前场景匹配的已有方法；
2. 改造方法输入，使其适配 skill instruction、skill script 或组合 skill；
3. 在目标检测器上批量测试；
4. 分析哪些方法有效，哪些方法无效，以及失效原因。

## 五、待确认问题

1. 当前项目最终采用的攻击定义应如何命名：adversarial attack、LLM judge evasion，还是 skill maliciousness detector evasion。
2. 目标检测器的输入边界是什么：只检测 instruction，还是同时检测 markdown、代码、脚本和外部输入。
3. 多 skill 组合攻击是否纳入主实验。
4. 邮件等外部输入触发的间接指令攻击是否作为独立场景。
5. 现有 jailbreak 相关内容是否作为对比方向写入背景部分。

## 六、下一步

1. 先从 adversarial attack 和 LLM judge 欺骗方向做文献调研。
2. 优先选择有代码的论文复现或改造。
3. 将已有攻击方法迁移到 skill instruction、skill script 和组合 skill 三类输入上。
4. 用当前检测工具测试绕过效果。
5. 根据实验结果更新问题定义和论文写法。
