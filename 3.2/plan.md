# Agent Skills 后续研究执行计划（v2）

更新时间：2026-03-09  
依据资料：`3.2/minutes.md`、`3.2/3_原文 (1).txt`、`skills_data/sampled_skills_1000/*`、现有分析脚本与已有结果

## 1. 目标与原则

本阶段目标不是把会议中的 4 个方向同时做成完整大工程，而是基于仓库现状，优先形成一条：

`可复现 -> 可人工校准 -> 可统计分析 -> 可写论文`

的方法链路。

执行原则：

1. 先修复数据与脚本链路，再扩展分析问题。  
2. 优先做当前样本最能支撑的主线问题，避免在弱证据问题上过度展开。  
3. 规则抽取必须以高精度、可解释为先，不追求一开始高召回。  
4. 所有统计结果默认先做描述性分析，再谨慎进入回归或因果式表述。  

## 2. 数据口径与当前判断

### 2.1 主数据口径

1. 主样本：`skills_data/sampled_skills_1000/`  
2. 样本规模：`popular=1000`，`random=1000`  
3. 元数据总表：`skills_data/skills_metadata.json`（`total_skills=184822`）  

### 2.2 本阶段比较范围

本阶段统计分析只比较两组：

1. `popular 1000`
2. `random 1000`

`anthropic_official_skills` 暂不单独纳入比较，也不作为独立 cohort 展开统计分析。

### 2.3 当前仓库能力边界

当前仓库已经具备：

1. 采样与下载脚本  
2. `SKILL.md` 与文件结构的粗粒度特征抽取  
3. 人工复核候选集生成  
4. `popular vs random` 的已有描述性结果  

当前仓库尚不具备：

1. 系统化的 skill-to-skill 依赖边抽取  
2. 依赖机制分类与风险传导建模  
3. 脚本行为语义级分类  
4. `SKILL.md` 细粒度 codebook 与一致性评估  

### 2.4 研究优先级判断

基于会议纪要与样本现状，当前优先级调整为：

1. 主线：WP3 `SKILL.md` 细粒度内容方法学  
2. 主线：WP4 附属文件深度分析  
3. 探索线：WP1 依赖证据审计  
4. 条件触发：WP2 依赖机制与风险传导  

原因：

1. 会议纪要中，依赖关系本身仍处于“先看有没有”的阶段。  
2. 本地样本中存在少量 cross-skill 痕迹，但误报风险高，不适合直接上完整依赖图。  
3. `SKILL.md` 结构化程度、约束/触发、附属文件行为，是当前仓库最容易做出稳健结果的方向。

## 4. 修订后的工作包设计

## WP1：Skills 依赖证据审计（探索性）

研究问题：在当前 2000 个已下载 skills 中，是否存在可信的 skill-to-skill 调用/依赖证据？

这里不再一开始定义“完整依赖图”，而是先做高精度证据审计。

### 方法步骤

1. 定义“可接受的依赖证据”  
优先只保留以下高精度证据：
   - E1：相对链接或路径直接指向另一个 `SKILL.md`
   - E2：文本中明确出现 `see/use/invoke ... skill`
   - E3：脚本或命令中直接引用 `~/.claude/skills/...`
   - E4：同仓库中明确声明调用顺序或终态调用另一个 skill

2. 排除高频误报  
重点排除：
   - `$ARGUMENTS`、`$PROJECT`、`$UTILS` 一类占位变量
   - 普通软件依赖语义中的 `dependency/dependencies`
   - 仅提到工具、技术名词，但未指向 skill 实体的文本

3. 建立候选词典  
词典仅包含：
   - 本地 2000 skill 的 `skill_id`
   - `name`
   - repo 内相邻 skill 名
   - 官方 skill 名列表

4. 高精度抽取  
按证据类型输出候选边，不追求一次性高召回。

5. 分层人工复核  
对每类证据抽样复核，计算 precision。  
若某类规则 precision `< 0.85`，则不进入正式统计。

### 输出

1. `dependency_candidates.csv`  
字段：`source, target_candidate, evidence_text, evidence_type, local_path, confidence`
2. `dependency_audit.md`  
3. `dependency_review_sample.csv`  

### 成功标准

满足以下任一条件即可：

1. 发现足够多的高精度依赖证据，可进入 WP2  
2. 证明当前生态中 skill-to-skill 依赖稀疏、弱标准化或极少显式化  

第二种结果同样是有效研究发现。

## WP2：依赖机制与风险传导（条件触发）

启动条件：仅当 WP1 发现“数量足够且 precision 达标”的依赖边时启动。

研究问题：若依赖存在，它是如何实现的？是否会引入兼容性与风险传导问题？

### 方法步骤

1. 机制分类  
将经人工确认的依赖边分为：
   - M1 文档级建议调用
   - M2 指令级显式调用
   - M3 路径级/脚本级调用
   - M4 资源级复用

2. 耦合信息抽取  
只抽取可见、可验证的信息：
   - 固定路径耦合
   - repo 内部相对引用
   - 分支或版本文本提示
   - 运行前置条件

3. 风险传导分析  
   不直接声称“漏洞已传导”，而是定义为：
   - 上游 skill 含高风险脚本行为
   - 且被多个下游 skill 显式引用
   - 将其记为“潜在风险传导路径”

### 输出

1. `dependency_mechanisms.csv`  
2. `compatibility_findings.md`  
3. `potential_risk_propagation.md`  

### 停止条件

若 WP1 中确认边数量过少，则 WP2 不展开，改写为：

“当前样本中依赖关系稀疏，尚不足以支持稳定的机制分类与传导建模。”

## WP3：`SKILL.md` 细粒度内容方法学（主线）

研究问题：结构化表达、约束/触发、spec 形式是否与 `popular` 和 `random` 两组之间存在稳定差异？

### 方法步骤

1. 建立 codebook  
至少包含：
   - C1 约束/禁止语句
   - C2 触发条件语句
   - C3 错误处理/回退策略
   - C4 输入输出格式约束
   - C5 spec 表达形式  
     - 自然语言  
     - 结构化模板  
     - DSL  
     - 代码/伪代码  
     - 数学/逻辑表达  

2. 结构化程度评分  
建议仍保留 0-100 分，但改为“可审计分项”：
   - 触发条件（20）
   - 约束条件（20）
   - 步骤化流程（15）
   - 错误处理（15）
   - I/O 规范（15）
   - 示例质量（15）

3. 标注策略  
先做小样本双人盲标，再扩展：
   - popular 与 random 分层抽样
   - 双人标注
   - 冲突仲裁
   - Cohen’s Kappa，目标 `>=0.75`

4. 自动扩展  
人工标注稳定后，再把规则或模型扩展到更大样本。  
自动标注结果必须保留人工校验集。

5. 统计分析  
   分两层：
   - 描述统计：popular vs random 差异
   - 必要时补充简单稳健性检查

### 输出

1. `annotation_schema.md`  
2. `manual_labels.csv`  
3. `structuredness_score.csv`  
4. `method_results_skillmd.md`  

## WP4：附属文件深度分析（主线）

研究问题：附属文件复杂度、行为风险与支持材料丰富度如何分布？与 cohort 差异如何？

注：会议里提到“数据库分析”，但基于当前样本，应扩展为更稳妥的“数据资产分析”。

### 方法步骤

1. 脚本行为分类  
对脚本做操作级标签抽取：
   - 读
   - 写
   - 删
   - 执行外部命令
   - 网络访问
   - 凭据访问

2. 示例分析  
   区分两类示例：
   - `SKILL.md` 内嵌示例
   - 独立 `examples/` 或示例脚本  

统计：
   - 示例段数量
   - 示例代码块数量
   - 示例与步骤是否一致

3. 数据资产分析  
   不局限于数据库，统一识别：
   - `json/csv/jsonl`
   - 模板文件
   - 资源文件
   - 参考文档
   - 若存在 `sqlite/db` 再单列  

统计：
   - 文件类型
   - 文件数
   - 总大小
   - 目录复杂度

4. 联合分析  
   将 WP4 的脚本风险、示例丰富度、数据资产复杂度，与 WP3 的结构化分数联合，识别：
   - 高结构化-高能力-高风险
   - 高结构化-低风险
   - 低结构化-高风险
   - 低结构化-低风险

### 输出

1. `script_ops.csv`  
2. `example_stats.csv`  
3. `data_asset_stats.csv`  
4. `artifact_profile.md`  

## 5. 统计与论文表述约束

### 5.1 不宜直接做的表述

以下表述默认禁止直接写入论文主结论：

1. “popular skill 更好”  
2. “结构化导致更高流行度”  
3. “依赖一定导致漏洞传导”  

### 5.2 可接受表述

可使用更稳健的表述：

1. “popular 与 random 在结构化表达上呈现可观察差异”  
2. “当前样本中发现了有限但明确的 cross-skill 证据”  
3. “某类依赖模式可能形成潜在风险传导路径”  

## 6. 推荐执行顺序

1. Phase 0：复现性修复  
2. WP3：先完成 codebook、小样本标注与 structuredness score  
3. WP4：完成脚本行为、示例与数据资产分析  
4. WP1：完成高精度依赖证据审计  
5. WP2：仅在 WP1 达到启动条件时展开
