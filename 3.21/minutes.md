# 3月21日会议纪要：Agent Skill 安全研究讨论

## 会议主题
基于大模型 Agent Skill 的安全问题（重点探讨了恶意注入、静态扫描工具的有效性、以及相关工作的推进方向）。

## 核心讨论内容

### 1. Agent Skill 安全的现状与最新进展 (Tao 汇报)
*   **企业动向**：各大公司（如 OpenAI、微软）逐渐将防御重点从输入过滤转向底层（Runtime/Sandbox）的安全隔离。OpenAI 将 prompt injection 视为社会工程学问题，强调用户确认与底层防御拒绝机制。
*   **主流机制分类**：当前 Agent Skill Security 的研究和实践主要分三条主线：
    1.  **Skill Artifact Scanning**：静态扫描与检测（类似传统软件测试与分析，结合大模型语义扫描）。
    2.  **Runtime Protection**：Agent 底层运行时的执行保护与权限控制。
    3.  **Registry Governance**：类似 App Store 的平台注册、准入与上架审查机制。
*   **现有攻击研究反馈**：提到一个名为 "Skill Inject" 的基准测试，显示恶意指令成功注入率高达 80%。恶意行径包括数据外泄、破坏性操作、勒索及凭据窃取等。目前单纯依靠 Prompt 层面的过滤和防护效果有限。

### 2. 关于检测机制的难点与探索
*   由于 Skill 的指令基本由极其非结构化的“自然语言”编写，传统的静态分析很难直接应用。
*   团队探讨了借用传统软件工程中“Source and Sink（数据流分析）”的思想来检测越权的数据流动（如恶意下载或数据外传），难点在于如何从自然语言指令中准确识别这些行为及其意图的合理性。
*   目前存在两层防御博弈：坏人可以通过伪装正常的 Skill（或者使用相同名称）进行注册欺骗（身份混淆/供应链攻击），而现存部分扫描工具规则粗糙，易产生误报。

### 3. 研究方向的分歧与决策：安全 (Security) vs. 软件工程 (Software Engineering)
会议中讨论了两个推进方向：
*   **软工方向**：分析并总结出一套编写 Skill 的 “Best Practice（最佳实践/规范）”。通过让开发者遵守固定的结构和规范，使第三方的安全审计和扫描工具更容易工作。不过难点在于当前缺乏官方推荐的标准范式，全凭经验，评判好坏的界限模糊。
*   **安全方向（被选定）**：针对现有的各类 Skill 检测工具（Scanners）做一个系统性的评估基准（Benchmark）。因为目前该领域正处于快节奏阶段，大家更关注安全性。

## 结论与下一步工作 (Action Items)
考虑到安全方向的紧迫性和发文潜力，团队**优先推进安全方向（Security）**的研究。

**Tao 的下一步工作安排：**
1.  **文献精读**：深入阅读并准备在下周会议上汇报《Agent Skills in the Wild: An Empirical Study of Security...》以及关于 `Skill Inject` 的论文，看看现有研究已经做到什么程度。
2.  **调研现有检测工具（Scanner）**：梳理目前可用的 Scanner，看看它们到底能跑通多少检测。
3.  **构建 Benchmark 设想**：尝试设计/构建一个包含各类攻击模式（例如：只下载不外传、既下载又外传等）的对抗性数据集。目的不是单纯绕过当前工具，而是系统性地对当前各类 Scanner 防御工具进行 Benchmark 评估，分析它们的漏报与不足，试图挖掘出更深层的 pattern。
