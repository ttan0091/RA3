# AI Skill 安全检测 Benchmark 构建实施计划

**基于 3.24 会议纪要，结合仓库已有调研成果和学界/业界最新进展**

---

## 0. 现状评估

### 已有基础（仓库内）
- 2000 条 skill 样本（popular 1000 + random 1000），已完成结构化分析、依赖审计、制品分析
- Cisco Skill Scanner 方法论逆向分析完成（静态签名 + 行为数据流 + LLM 语义分析 + 元分析四层架构）
- 文献综述覆盖 SkillFortify、SkillScan、ClawHavoc 事件、Clawdrain token drain 攻击等
- 已有 WP3 人工标注 pilot（12 条，kappa 高一致性）

### 外部最新进展（2026 年 1-3 月）
- **ClawHavoc 事件**：ClawHub 中确认 1,184 个恶意 skill，Koi Security 发现 341 个，Cisco AI Defense 开源了 [skill-scanner](https://github.com/cisco-ai-defense/skill-scanner)
- **SkillFortify**（[GitHub](https://github.com/varun369/skillfortify)）：形式化分析框架，F1=96.95%，precision=100%，支持 22 个框架（MCP、LangChain、CrewAI 等）
- **SkillScan**（[skillscan.dev](https://skillscan.dev/)）：多层检测框架，precision=86.7%，recall=82.5%
- **AgentShield benchmark**：首个开放 benchmark，537 test cases 测试 6 款商用安全工具，发现 tool abuse 检测普遍薄弱
- **大规模实证**（[arxiv 2602.06547](https://arxiv.org/html/2602.06547v1)）：98,380 skills，632 漏洞映射到 13 种攻击技术和 6 个 kill chain 阶段
- **MCP 安全问题爆发**：tool poisoning、sampling 攻击、间接 prompt injection 经 MCP 放大
- **Cisco DefenseClaw**：2026-03-27 开源，结合 OpenShell 约束 + AI Defense 验证

### 关键差距（我们的机会）
1. 现有 scanner 之间结果差异巨大（flagging rate 从 0.52% 到 41.93%），缺乏统一的对抗性 benchmark 来量化谁在说真话
2. 现有 benchmark（AgentShield 537 cases）规模小，且未包含"加固规避版本"的对抗样本
3. 无人系统性地测试 scanner 面对主动规避（obfuscation、remote loading、语态变换）时的退化程度
4. 形式化方法（SkillFortify）与启发式方法（Cisco Scanner）的互补性未被量化

## 1. 阶段一：漏洞分类体系与攻击模式库

> 对应会议 Action A（负责人：Tao）

### 1.1 构建漏洞分类标准

**输入**：已有文献（仓库内 3.11 调研报告、News.md、review.md）+ 最新论文

**分类维度**（建议采用三轴分类法）：

| 维度 | 说明 | 示例值 |
|------|------|--------|
| **攻击技术** | 具体利用手法 | Prompt Injection（直接/间接）、Privilege Escalation、Data Exfiltration、Token Drain、Tool Poisoning、Dependency Hijack |
| **Kill Chain 阶段** | 攻击在哪个环节生效 | 安装期、加载期、运行期、通信期 |
| **载体层级** | 恶意指令藏在哪里 | SKILL.md 文本层、脚本/代码层、依赖引用层、元数据层、外部资源层 |

**具体操作**：
1. 从 [arxiv 2602.06547](https://arxiv.org/html/2602.06547v1) 的 13 种攻击技术出发，与 OWASP LLM Top 10（LLM07 Insecure Plugin Design）交叉对齐
2. 补充 MCP 特有攻击面：tool poisoning via description（[Palo Alto Unit42](https://unit42.paloaltonetworks.com/model-context-protocol-attack-vectors/)）、sampling abuse、conversation hijacking
3. 补充经济型攻击：Clawdrain 式 token drain（6-9x 放大）
4. 融合安卓权限模型视角：将 skill 的危险能力映射为类似 `INTERNET`、`READ_CONTACTS`、`WRITE_EXTERNAL_STORAGE` 的权限声明

**交付物**：`《Skill 安全漏洞分类手册 v1》`，包含分类表 + 每类的典型 case 描述 + 已知检测难点

### 1.2 攻击模式提炼

从文献和真实事件中提炼 **10 种隐藏攻击模式**：

| # | 攻击模式 | 来源/灵感 | 检测难度 |
|---|---------|----------|---------|
| 1 | 直接 prompt injection（SKILL.md 中明文恶意指令） | ClawHavoc 基础版 | 低 |
| 2 | 间接 prompt injection（通过外部 URL/文件注入） | MCP GitHub issue 投毒 | 中 |
| 3 | 语态伪装（过去式/假设语气描述恶意行为） | 会议纪要提到 | 中 |
| 4 | 条件触发（仅在特定上下文/时间激活） | Sleeper agent 模式 | 高 |
| 5 | 工具描述投毒（tool description 中嵌入隐藏指令） | Palo Alto Unit42 | 高 |
| 6 | 依赖劫持（引用已废弃的 GitHub repo） | 2602.06547 supply-chain | 中 |
| 7 | 数据渗出（伪装为正常 API 调用的数据外传） | Data Thief archetype | 中 |
| 8 | Token drain（通过 tool-calling chain 放大消耗） | Clawdrain 论文 | 中 |
| 9 | 权限提升链（组合多个单独安全的 tool 实现越权） | Confused deputy | 高 |
| 10 | 元数据欺骗（虚假下载量/评分 + 品牌冒充） | ClawHavoc 54.1% templated | 低 |

## 2. 阶段二：对抗性 Benchmark 库构建

> 对应会议 Action B（负责人：说话人4 & 团队）

### 2.1 Benchmark 设计原则

1. **双版本原则**：每种攻击模式必须包含"原始版本"（容易检测）和"加固规避版本"（尝试绕过 scanner）
2. **ground truth 标注**：每条 case 需标注攻击向量、预期检测点、kill chain 阶段、危害等级
3. **覆盖全载体层级**：不能只做 SKILL.md 文本层，必须覆盖脚本层、依赖层、元数据层
4. **包含良性对照组**：加入功能相似但无恶意的 skill 作为 negative samples，测试 false positive rate

### 2.2 构建流程

```
Step 1: 从 10 种攻击模式出发，每种编写 2 个原始版本（共 20 个）
Step 2: 对每个原始版本，编写 2 个加固规避变体（共 40 个）
Step 3: 编写 30 个良性对照 skill（含容易被误判的正常行为，如正常数据导出、合法 API 调用）
Step 4: 全部标注（攻击向量、检测点、难度等级、预期各 scanner 表现）
Step 5: 内部交叉审核（至少两人 blind review，计算 kappa）
```

**最终规模**：约 **90 条** 高质量标注 case（60 malicious + 30 benign）

### 2.3 加固规避技术清单

编写规避版本时，系统性地应用以下技术：

| 规避技术 | 说明 | 针对哪类 scanner |
|---------|------|-----------------|
| 语态/人称变换 | "曾经有用户需要导出所有文件..." | 基于关键词的静态规则 |
| 指令分片 | 将恶意指令拆成多个无害片段，由 LLM 重组 | 单句级语义分析 |
| Base64/编码混淆 | 将 payload 编码后要求 agent 解码执行 | 静态签名匹配 |
| 远程加载 | SKILL.md 中仅包含 URL，恶意内容托管在外部 | 纯静态分析（无网络访问） |
| 良性外壳包装 | 大量正常内容中夹带少量恶意指令 | 全文 LLM 判断（context dilution） |
| 工具链组合 | 单个 tool 调用无害，组合后产生恶意效果 | 单步行为分析 |
| 时间/条件门控 | `if date > X` 或 `if user_role == admin` 才激活 | 无运行时上下文的静态分析 |

### 2.4 Benchmark 数据格式

```yaml
id: "MAL-PI-003"
name: "Stealth Data Exfil via API Helper"
category: "data_exfiltration"
attack_pattern: "indirect_prompt_injection"
variant: "evasion_v1"  # original / evasion_v1 / evasion_v2
kill_chain_phase: "runtime"
carrier_layer: "skill_md"
severity: "high"
evasion_techniques: ["persona_shift", "benign_wrapper"]
expected_detection:
  cisco_scanner: false  # 预期能否检出
  skillfortify: true
  skillscan: false
ground_truth_label: "malicious"
description: "表面是 API 调用助手，实际将用户对话内容编码后通过合法 API endpoint 外传"
detection_hints:
  - "检查是否将用户输入作为参数传递给外部 HTTP 请求"
  - "检查是否存在 base64 编码操作后接网络请求"
files:
  - skill.md
  - run.py  # 如果涉及脚本层
```

## 3. 阶段三：主流工具部署与交叉横评

> 对应会议 Action C（负责人：Xiao & Tao）

### 3.1 待评估工具清单

| 工具 | 类型 | 获取方式 | 优先级 |
|------|------|---------|--------|
| [Cisco Skill Scanner](https://github.com/cisco-ai-defense/skill-scanner) | 开源，多层分析 | GitHub 直接 clone | P0 |
| [SkillFortify](https://github.com/varun369/skillfortify) | 开源，形式化验证 | GitHub 直接 clone | P0 |
| [SkillScan](https://skillscan.dev/) | SaaS/开源，多层检测 | API 或联系作者 | P1 |
| Cisco DefenseClaw | 开源（2026-03-27） | GitHub（即将发布） | P1 |
| Snyk Agent Scanner | 商用，LLM-as-judge | 联系获取试用 | P2 |

### 3.2 评估指标

```
核心指标：
- Precision（误报率的反面）
- Recall（漏报率的反面）
- F1-score
- 按攻击模式的 recall breakdown（哪类攻击最容易漏？）
- 按规避技术的 recall degradation（加固后检出率下降多少？）

辅助指标：
- 扫描耗时（单 skill 平均）
- 是否需要网络访问/LLM API
- 是否支持脚本层/依赖层分析
- 误报模式分析（哪类正常行为最容易被误判？）
```

### 3.3 实验设计

```
实验 1：基线检测能力
  - 输入：20 个原始版本恶意 skill + 30 个良性 skill
  - 测量：各工具的 precision / recall / F1

实验 2：对抗鲁棒性
  - 输入：40 个加固规避版本
  - 测量：recall degradation（相比原始版本下降多少）
  - 分析：按规避技术分组，找出各工具的致命弱点

实验 3：互补性分析
  - 将所有工具的检测结果取并集/交集
  - 计算：组合后的 precision / recall 上界
  - 分析：哪些组合策略能最大化 F1？

实验 4：载体层级覆盖度
  - 按 SKILL.md / 脚本 / 依赖 / 元数据分层统计各工具覆盖情况
  - 找出所有工具都覆盖不到的盲区
```

### 3.4 失效原因分析框架

对每个漏报 case，记录：

| 字段 | 说明 |
|------|------|
| tool | 哪个工具漏报 |
| case_id | 漏报的 case |
| root_cause | 失效根因（规则缺失 / 语义理解失败 / 架构盲区 / context window 限制） |
| fixability | 该工具框架内是否可修复（easy / hard / architectural） |

## 4. 阶段四：成果整合与论文输出

### 4.1 核心贡献点（论文卖点）

1. **首个对抗性 AI Skill 安全 Benchmark**：不是随机数据集，而是系统性覆盖 10 种攻击模式 x 多种规避技术的标注数据
2. **首次量化主流 scanner 的对抗鲁棒性退化**：回答"现有工具面对刻意规避时表现如何"
3. **互补性发现**：形式化方法（SkillFortify）vs 启发式方法（Cisco）vs 语义方法（SkillScan）各自的强弱区
4. **安全盲区地图**：所有工具都检测不到的攻击模式 + 规避技术组合

### 4.2 可选扩展方向

- **权限声明机制原型**：基于漏洞分类，设计一个类 Android Manifest 的 skill 权限声明 schema，并验证其对检测的增益
- **运行时检测补充**：对静态 scanner 检测不到的 case，探索运行时 guardrail（如 NeMo Guardrails / Microsoft Prompt Shields）的互补效果
- **大规模验证**：将 benchmark 中表现最好的 scanner 组合应用于仓库内已有的 2000 条 skill 样本，报告 wild 环境中的检出率

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 部分工具未开源或 API 受限 | 横评不完整 | 优先保证 Cisco Scanner + SkillFortify（已开源）；其他工具联系作者或降级为文献对比 |
| 加固规避版本编写质量不够"对抗性" | Benchmark 区分度不足 | 引入 red team 思维，参考 ClawHavoc 真实样本的规避手法；邀请安全背景成员审核 |
| Scanner 运行环境依赖复杂 | 部署延迟 | 提前一周开始环境搭建；准备 Docker 化部署方案 |
| 良性 skill 被误判比例过高导致 precision 指标失真 | 评估结论可靠性下降 | 增加良性对照组多样性；包含"易误判"的边界 case |
| 论文定位与已有 AgentShield benchmark 重叠 | 新颖性不足 | 强调"对抗性"（双版本设计）和"规避鲁棒性退化量化"作为核心差异点 |
