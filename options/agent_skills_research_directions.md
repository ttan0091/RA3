# Agent Skills 实证研究方向：综合深度分析

> **背景**：Agent Skills 是 2025-2026 年 LLM Agent 领域的新兴核心抽象层——可组合的模块化知识包（通常以 `SKILL.md` 为载体，含 YAML 元数据、逐步指令、脚本、资源文件等），Agent 可在推理时按需加载，实现**无需重训的动态能力扩展**。这一范式正处于爆发期（GitHub 6万+ 星，skillsmp.com 27万+ skills），但**系统性实证研究极度稀缺**。
>
> 本文综合多个来源的分析，提炼出 **7 个高价值实证研究方向**，按"可行性 × 创新性 × 影响力"排序，附带详细的研究问题、数据/方法论设计和投稿建议。

## 二、各方向详细分析

---

### 方向 1：Agent Skills 生态大规模特征分析（Mining Study）

> [!IMPORTANT]
> 这是与现有数据基础设施**最匹配**、启动成本**最低**、且学术价值**极高**的方向。目前**无任何论文**对 27万+ skills 生态进行过系统性实证分析。

#### 核心定位

对 skillsmp.com 27万+ 社区 Skills 进行**首次系统性大规模挖掘与特征刻画**——类似于此前学术界对 npm/PyPI/Docker Hub/GitHub Actions 生态所做的经典 mining study。

#### 研究问题（RQs）

| RQ | 具体问题 | 分析方法 |
|:---|:---------|:---------| 
| RQ1 | **功能分布**：Skills 主要集中在哪些领域？长尾分布如何？ | NLP 主题建模（LDA / BERTopic），手动编码验证 |
| RQ2 | **结构复杂度**：多少 Skills 包含脚本/模板？平均 token 数？元数据规范遵循度？ | 静态分析、规范校验脚本 |
| RQ3 | **质量分布**：高星 vs 低星 Skills 有什么结构性差异？描述清晰度、步骤完整性？ | 质量评分框架（人工 + LLM 混合编码） |
| RQ4 | **重复与克隆**：有多少 Skills 是复制粘贴或轻度修改？ | 文本克隆检测（SimHash / D-ECT） |
| RQ5 | **创作者生态**：头部创作者贡献占比？组织 vs 个人？ | Lorenz 曲线、Gini 系数、社区网络分析 |

#### 数据与方法

```
Phase 1: 全量元数据统计分析（RQ1, RQ5）
  → 27万+ metadata 记录（已有 skills_metadata.json）
Phase 2: 分层采样 + 内容深度分析（RQ2, RQ3）
  → 扩大 SKILL.md 下载至 2000-5000 份
Phase 3: 克隆/重复检测（RQ4）
  → 全文相似度矩阵
Phase 4: 质量评估框架构建与验证
  → 人工编码 + Cohen's Kappa
```

#### 参考论文范式
- GitHub Actions 的 Mining Study（ICSE 2021）
- npm/PyPI 生态系统分析（FSE/ESEC 系列）
- Docker Hub 镜像大规模分析（MSR 系列）

#### 投稿目标
**MSR**（Mining Software Repositories）、**ESEC/FSE**、**ICSE-SEIP**

### 方向 2：Agent Skills 安全漏洞大规模分析（Security Study）

> [!WARNING]
> 已有一篇 arXiv 论文（2601.10338）对 31,132 个 skills 进行了安全分析，发现 **26.1% 含至少一种漏洞**。需要在此基础上进行**差异化**——扩大规模、增加动态验证维度、或聚焦治理框架的实证评估。

#### 核心定位

在已有静态分析基础上，进一步进行**动态验证**和**治理框架实证**，或扩展安全分析的规模与维度。

#### 已知关键发现（来自现有研究）

| 发现 | 数据 |
|:-----|:-----|
| 含漏洞 skills 比例 | **26.1%**（31,132 个样本） |
| 最常见漏洞类型 | 数据泄露 13.3%，权限提升 11.8% |
| 脚本 vs 纯指令 | 含可执行脚本的 skill 漏洞概率高 **2.12 倍**（OR=2.12） |
| 疑似恶意 | 5.2% |
| 相关事件 | ClawHub 市场发现 1,184 个恶意 skill，91% 利用 Prompt Injection |

#### 差异化研究问题

| RQ | 具体问题 |
|:---|:---------|
| RQ1 | 现有分析提出的 **4 级信任层级治理框架**（静态分析 → 语义分类 → 沙箱行为 → 权限校验）在实践中效果如何？ |
| RQ2 | Skills 的**权限声明**是否遵循最小权限原则？过度权限声明有多普遍？ |
| RQ3 | **Prompt Injection** 在 skill 指令中的模式与成功率？恶意 skill 能否劫持 Agent？ |
| RQ4 | Skills 依赖的**外部资源**（API、URL、npm 包）存在供应链攻击风险吗？ |
| RQ5 | **纵向跟踪**：已删除的 17.3% skills 中有多少是恶意的？漏洞演化趋势？ |

#### 方法论

1. **静态分析**：对含脚本 skills 运行 CodeQL / Semgrep / Bandit
2. **动态沙箱验证**：在 Docker 容器中执行 skill，监控网络行为、文件访问、数据外传
3. **权限合理性分析**：解析 YAML `tools` 字段，与任务描述比对权限必要性
4. **Prompt 注入检测**：构建 adversarial prompt 模式库，检测 skill 指令中的注入模式
5. **供应链审计**：提取外部依赖，交叉比对 CVE / npm audit
6. **可扩展**：复用已开源的 **SkillScan** 工具（86.7% precision）

#### 投稿目标
**CCS**、**USENIX Security**、**S&P**（安全方向），或 **ICSE/FSE**（SE 安全方向）

### 方向 3：Skills 有效性与设计特征实证评估（Effectiveness Study）

> [!NOTE]
> SkillsBench（arXiv:2602.12670）已证明 *curated skills > no skills > self-generated skills*。但**它只用了 86 个精心设计的 task-skill 对**，没有分析社区"野生" skills 的有效性，更没有回答"**怎样的 skill 更有用**"。

#### 与 SkillsBench 的差异化

| 维度 | SkillsBench | 本方向（差异化） |
|:-----|:------------|:-----------------|
| 数据源 | 86 个 curated skills | 社区实际创建的 "in-the-wild" skills |
| 分析粒度 | 有/无 skills 的 pass rate 对比 | 分析 skill 内部**哪些设计特征**驱动了效果 |
| 领域覆盖 | 11 个预定义 domain | 来自 skillsmp.com 的自然领域分布 |
| 核心贡献 | 证明 skills 有用 | 回答**怎样的 skills 更有用**（可操作的设计指南） |

#### SkillsBench 关键实验数据（背景）

| 指标 | 数据 |
|:-----|:-----|
| 人工 skill 平均提升 | **+16.2pp** |
| 自生成 skill 平均效果 | **-1.3pp**（负面） |
| 领域差异 | 医疗 +51.9pp，制造 +41.9pp，SWE 仅 +4.5pp |
| 最佳 skill 数量 | **2-3 个**精简模块效果最优 |
| 小模型 + skill vs 大模型裸奔 | Claude Haiku 4.5 + skill (27.7%) > Claude Opus 4.5 无 skill (22.0%) |
| 极端案例 | Mario-coin-counting: 2.9% → 88.6%（+85.7pp） |

#### 研究问题

| RQ | 具体问题 |
|:---|:---------|
| RQ1 | Skill 的**长度**（token 数）与任务成功率之间是什么关系？是否存在"甜蜜点"？ |
| RQ2 | Skill 中包含**可执行脚本** vs 纯指令文本，对任务可靠性的影响如何？ |
| RQ3 | Skill 的**结构化程度**（步骤编号、错误处理、输入输出示例）与效果的相关性？ |
| RQ4 | **"生态级"（低质量，自动选择）vs "实验室级"（精心设计）**skills 的实际差距有多大？ |

#### 实验设计

1. 从 skillsmp.com 分层采样 200-500 个真实 skills
2. 为每类 skill 设计 3-5 个评估任务（手动 + LLM 辅助生成）
3. 控制变量实验：相同任务下使用 {无 skill, 原始 skill, 精简 skill, 扩展 skill}
4. 多模型测试：Claude 3.5 Sonnet, GPT-4o, 开源模型（Llama 3, Qwen 2.5）
5. 多维度指标：任务完成率、token 效率、输出一致性、首次正确率
6. 可复用 SkillsBench 的 **Harbor 容器化框架**（Docker 隔离 + 确定性验证）

> [!TIP]
> 先在开源模型上大规模筛选特征，再在商业模型上做验证性精细实验，可大幅降低 API 成本。

### 方向 4：技能获取方式的比较实证（Acquisition Comparison Study）

#### 核心定位

现有 survey（arXiv:2602.12430）将 skill 获取分为 4 大类（人工编写、RL 优化、自主发现、组合合成），但**缺乏头对头实证对比**——在同一任务集上，哪种方式最可扩展？

#### 研究设计

**在同一任务集上比较 4 种获取方式：**

| 获取方式 | 代表方法 | 关键特征 |
|:---------|:---------|:---------|
| 人工编写 | Expert 撰写 SKILL.md | 质量高、成本高、不可扩展 |
| RL 优化 | SAGE 式 RL with skill library | 自动化、需奖励信号 |
| 自主发现 | SEAgent 式经验总结 | 无需人工、质量不稳定 |
| 组合合成 | Agentic Proposing | 组合已有原子 skill、中间地带 |

#### 测量维度

- **初始性能**：首次使用的任务成功率
- **可复用性**：跨任务迁移效果
- **维护成本**：skill 需多频繁更新
- **Token 效率**：skill 消耗的上下文长度
- **Phase transition**：skill 库过大后准确率骤降的临界点

#### 方法
使用 AppWorld / OSWorld 等公开环境，控制 skill 库大小，观察规模效应。

### 方向 5：技能组合与编排的协同 vs 干扰实证（Composition Study）

#### 核心定位

SkillsBench 发现 **2-3 个 skill 效果最佳**，更多反而下降。这背后的机制是什么？

#### 研究问题

| RQ | 具体问题 |
|:---|:---------|
| RQ1 | 2-10 个 skills 的组合效应曲线是什么形状？ |
| RQ2 | 原子 skill 的效果能否预测复合性能？ |
| RQ3 | 冲突检测：两个 skill 何时会互相矛盾？ |
| RQ4 | 自动编排 vs 手动编排，效果差异如何？ |

#### 方法论

- 在 SkillsBench 任务上做 **factorial 设计实验**
- 使用**图模型**建模 skill 之间的依赖/冲突关系
- 测量 context 窗口利用率与性能的关系

#### 预期贡献

建立技能"化学反应"理论框架——哪些 skill 协同增效，哪些互相干扰，以及最优编排策略。

### 方向 6：跨平台互操作性实证（Interoperability Study）

#### 核心定位

评估 Skills 在不同 Agent 平台（Claude Code、Gemini CLI、Cursor、Codex、Goose 等）之间的**实际可移植性**。当前 skill 高度依赖特定框架，迁移失败率未知。

#### 研究问题

| RQ | 具体问题 |
|:---|:---------|
| RQ1 | 同一 Skill 在不同平台上的**功能等价性**如何？ |
| RQ2 | 平台特定扩展（如 Claude 的 `allowedTools`）对互操作性的影响？ |
| RQ3 | agentskills.io 标准草案的**实际采纳度**和兼容性差距？ |

#### 方法

- 选取 100+ 高质量 skills
- 在 Claude Code、Gemini CLI、AutoGen、LangGraph 等框架间迁移
- 量化成功率、性能衰减、所需适配工作量
- 自动化迁移脚本 + 人工审计
- 测量 MCP 兼容性

#### 预期贡献

首个可移植性基准，推动社区标准化（类似 SKILL.md v2.0 提案）。

### 方向 7：评估指标革新与认识论胜任力（Evaluation Methodology Study）

> [!NOTE]
> 传统的"黑盒式"评估（仅看最终答案对不对）已无法满足对 Agent 技能的深度理解。Agent 经常在推理逻辑完全错误的情况下偶然得到正确答案。

#### 核心定位

超越 pass rate，建立**多维度、过程导向**的 skill 效果评估体系。

#### 关键评估维度

**认识论胜任力指标**（来自 SeekBench 框架）：

| 指标 | 定义 | 典型失败模式 |
|:-----|:-----|:-------------|
| 扎实度（Groundedness） | 每一步推理是否有观测到的证据支持 | 脱离检索证据，转而依赖模型幻觉 |
| 恢复能力（Recovery） | 初始搜索失败时，能否自适应调整策略 | 陷入无效查询死循环 |
| 校准度（Calibration） | 能否正确评估当前证据是否足够 | 过早终止搜索，或对矛盾证据过于自信 |

**过程质量指标**（来自 Agent GPA / DeepEval）：

| 指标 | 定义 |
|:-----|:-----|
| Goal Fulfillment | 目标达成度 |
| Plan Quality | 推理结构合理性 |
| Plan Adherence | 执行对计划的忠实度 |
| Logical Consistency | 跨步骤无矛盾 |
| Execution Efficiency | 资源经济性 |

#### Skill 专属质量指标（需新建）

| 指标 | 定义 |
|:-----|:-----|
| Reusability | 跨任务迁移成功率 |
| Composability | 与其他 skill 组合后的增益/衰减 |
| Maintainability | 环境变化后 skill 仍有效的持久度 |
| Token Efficiency | 每提升 1pp 性能所需的额外 token |

#### 研究切入点

- 对比不同尺寸模型（7B vs 70B）在遭遇虚假证据时的校准误差差异
- 研究推理强化模型（如 Search-R1）在"证据不足识别"上的表现是否反而逊于基座模型
- 建立可被后续研究直接复用的**标准评估工具包**

## 三、综合推荐与实施策略

### 🔥 最强推荐组合：方向 1 + 方向 2（Mining + Security）

**理由：**

1. **数据复用最大化**：两个方向共享同一数据源（skillsmp.com），采集成本摊薄
2. **互相增强**：方向 1 的结构分析自然提供安全分析的输入特征
3. **学术空白清晰**：目前无任何论文对 27万+ skills 生态做过系统刻画
4. **可拆分为两篇论文**：
   - 方向 1 → MSR / ESEC/FSE（SE 社区）
   - 方向 2 → CCS / USENIX（安全社区）
5. **与团队能力匹配**：已有数据采集脚本和初步分析代码

### 🥈 次优推荐：方向 1 + 方向 3（Mining + Effectiveness）

如果有算力/API 预算，方向 3 的学术影响力最高——因为它直接回答**"什么样的 skill 更有用"**这个核心问题，对整个社区都有实操指导意义。

### 📋 建议的工作计划

```
Phase 1 (Month 1-2): 数据基础设施完善
  - 扩大 SKILL.md 下载规模至 5000+
  - 构建元数据统计分析 pipeline
  - 构建多维度质量分类编码体系

Phase 2 (Month 2-4): 方向 1 核心分析
  - 全量元数据统计（RQ1, RQ5）
  - 采样内容深度分析（RQ2, RQ3）
  - 克隆检测（RQ4）

Phase 3 (Month 3-5): 方向 2 或方向 3
  - 选项A（安全）: 脚本提取 → 静态扫描 → 权限分析 → Prompt注入检测
  - 选项B（有效性）: 分层采样 → 实验设计 → 控制变量 → 多模型测试

Phase 4 (Month 5-6): 论文撰写
  - 两篇论文并行撰写
```

## 四、与关键已发表工作的关系

### 与 SkillsBench 的关系

> [!TIP]
> SkillsBench 不是竞争对手，而是**互补关系**：
> - SkillsBench 回答"Skills 有没有用"（benchmark quantification）
> - 本研究回答"社区实际创建了什么样的 Skills"（ecological characterization）和"什么样的 Skills 更有用/更安全"（design guidelines）
>
> 在论文中应明确引用 SkillsBench 的发现，并指出本研究从 "in-the-wild" 视角的**独特互补贡献**。

### 与安全漏洞论文（2601.10338）的关系

该论文对 31,132 个 skills 做了静态安全分析。本研究可以在此基础上：
- **扩大规模**（27万+ vs 3万+）
- **增加动态验证**（沙箱行为监控）
- **纵向追踪**（漏洞演化趋势）
- **治理框架实证**（4级信任模型的因果验证）

### 与 SoK 调研论文（2602.12430 / 2602.20867）的关系

这些论文提供了理论框架（架构、获取分类、安全分类、挑战清单），但**缺乏实证数据支撑**。本研究填补的正是"theory ↔ evidence"之间的鸿沟。
